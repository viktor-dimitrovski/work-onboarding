# Intermediate Developer Assessment Pack

## For AI-era screening across K8S, .NET Core, C#, SQL, REST, Open Banking, automation, and payments

This is built to detect whether a candidate is truly **intermediate** and can work safely with AI, not just prompt AI and repeat output.

---

## 1. What to test in the AI era

Do not overvalue definitions.
Test these instead:

1. **Can they explain why**
2. **Can they spot risk**
3. **Can they debug broken logic**
4. **Can they verify AI-generated output**
5. **Can they make safe changes without breaking existing behavior**
6. **Can they reason across layers**
   example: token issue -> middleware -> gateway -> API -> DB -> logs

A real intermediate developer should usually be able to:

* explain tradeoffs, not only syntax
* debug step by step
* use AI as accelerator, not as substitute for judgment
* recognize security, reliability, and data consistency risks
* write small but correct automation/scripts
* test APIs and verify assumptions independently

---

## 2. Suggested assessment structure

### Part A — Oral / written technical interview

* 35 to 50 questions
* mix of short-answer and scenario questions

### Part B — Practical task

* 1 backend/API task
* 1 SQL/debugging task
* 1 automation or shell/cURL/Postman task

### Part C — AI-era validation

Give the candidate:

* AI-generated code
* AI-generated SQL
* AI-generated curl/Postman collection
* AI-generated explanation with hidden mistakes

Then ask them to validate and correct it.

---

## 3. Simple scoring model

Use this for each question:

### 0 — No real understanding

* vague
* memorized
* cannot explain consequences
* cannot give example

### 1 — Basic / junior-ish

* partial concept recognition
* weak reasoning
* misses operational/security implications

### 2 — Solid intermediate

* explains concept correctly
* gives practical example
* identifies common risks
* proposes workable debugging path

### 3 — Strong intermediate / approaching senior

* explains tradeoffs clearly
* connects multiple layers
* anticipates failure modes
* proposes safe implementation and verification steps

## 17. Practical exercises

## Exercise A — REST API verification

Give candidate:

* endpoint docs
* auth flow
* one broken endpoint behavior

Ask them to:

1. obtain token
2. call protected endpoint with cURL
3. reproduce same in Postman
4. identify whether issue is auth, payload, or business validation
5. write one automation check

**What good looks like**

* systematic
* uses headers/body carefully
* checks status and payload
* documents findings

Give candidate a small API with:

* wrong DI lifetime
* middleware order issue
* noisy exception handling
* inefficient EF query

Ask them to:

1. identify issues
2. explain impact
3. propose minimal safe fixes

* not over-refactoring
* understands runtime impact
* explains why fix is safe

## 18. Fast cheat detection in interview

Use these follow-ups:

### Follow-up 1

“Give me one real example from your work.”

Weak candidates collapse here.

---

### Follow-up 2

“What breaks if we do it the other way?”

Good intermediates can discuss consequences.

---

### Follow-up 3

“How would you verify that in logs, SQL, or cURL?”

This exposes whether they can operate, not just talk.

---

### Follow-up 4

“What are the top 2 risks in production?”

Very effective for separating theory from practical level.

## 20. Recommended pass profile for true intermediate

A solid intermediate should usually score:

* **.NET Core / C#**: 2
* **SQL**: 2
* **REST/security**: 2
* **automation/testing**: 2
* **K8S**: 1 to 2
* **payments/open banking**: 1 to 2 unless domain-experienced
* **AI-era validation maturity**: at least 2

If someone sounds polished but scores low on:

* idempotency
* middleware order
* DI lifetimes
* query validation
* API troubleshooting
* K8S service/probe debugging

then they are likely not truly intermediate.
